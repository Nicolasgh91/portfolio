Relaciones UML

—
Asociación
Vínculo simple. Ambos objetos pueden existir de forma independiente. Ej: Cliente compra Producto.
◇
Agregación
Relación todo–parte débil. La parte sobrevive sin el todo. Rombo blanco en el lado del todo. Ej: Equipo ◇ Jugadores.
◆
Composición
Relación todo–parte fuerte. La parte NO existe sin el todo. Ej: Casa ◆ Habitaciones.
▷
Herencia
La subclase "es un tipo de" la superclase. Flecha con triángulo vacío hacia el padre. Ej: Perro ▷ Animal.

UVA 3

Clases, SOLID y GRASP

Principios SOLID

S

Single Responsibility

Una clase debe tener una sola razón para cambiar. Una responsabilidad por clase.

O

Open / Closed

Abierta a la extensión, cerrada a la modificación. Extender sin romper.

L

Liskov Substitution

Las subclases deben poder reemplazar a sus superclases sin romper el programa.

I

Interface Segregation

No forzar a implementar métodos innecesarios. Interfaces pequeñas y específicas.

D

Dependency Inversion

Depender de abstracciones, no de implementaciones concretas.

Patrones GRASP (9 principios)

1 · Information Expert

Asignar la responsabilidad a la clase que tiene la información necesaria para cumplirla.

2 · Creator

La clase que usa/agrega/registra a B es responsable de crear instancias de B.

3 · Controller

Un objeto que no es de UI recibe y coordina los eventos del sistema.

4 · Low Coupling

Minimizar dependencias entre clases. Cambios en una no afectan a las demás.

5 · High Cohesion

Mantener clases enfocadas: las responsabilidades deben estar fuertemente relacionadas.

6 · Polymorphism

Usar interfaces/herencia para delegar comportamientos variables al tipo específico.

7 · Pure Fabrication

Crear una clase artificial cuando ninguna real es adecuada para mantener bajo acoplamiento.

8 · Indirection

Introducir un intermediario para evitar el acoplamiento directo entre dos clases.

9 · Protected Variations

Aislar puntos de variación con interfaces, wrappers o adaptadores para proteger el sistema de cambios.

Cursores (Iteradores)

Iterator

Todas las colecciones. Solo hacia adelante. Métodos: hasNext(), next(), remove().

ListIterator

Solo para List. Adelante y atrás. Agrega: hasPrevious(), previous(), add(), set().

Manejo de Excepciones

Throwable
  ├─ Error (no se manejan)
  │    ├─ OutOfMemoryError
  │    ├─ StackOverflowError
  │    └─ VirtualMachineError
  └─ Exception
       ├─ Checked (deben capturarse)
       │    ├─ IOException
       │    ├─ SQLException
       │    └─ FileNotFoundException
       └─ Unchecked (RuntimeException)
             ├─ NullPointerException
             ├─ ArithmeticException
             ├─ IndexOutOfBoundsException
             └─ IllegalArgumentException

throw vs throws vs Throwable

Elemento ¿Qué es? ¿Dónde? Ejemplo
throw Lanza una excepción Dentro del método throw new IOException()
throws Declara posibles errores Firma del método void x() throws Y
Throwable Superclase de excepciones Jerarquía de clases Throwable e = new Exception()

Estructuras try-catch-finally 

try {
    // código que puede fallar
    int resultado = 10 / 0;
} catch (ArithmeticException e) {
    // manejo específico
    System.out.println(e.getMessage());
} catch (Exception e) {
    // manejo genérico
} finally {
    // siempre se ejecuta (cierre de recursos)
}
